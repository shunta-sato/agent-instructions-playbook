# Structure-debt baseline ratchet (Issue #93) — ExecPlan

> This is a living document. Keep **Progress (WBS)**, **Decision log**, **Surprises**, and **Handoff** up to date.

## Purpose / Big Picture

- `scripts/check_structure.py` is state-based and correctly catches slow accretion, but it has no way to distinguish pre-existing, already-known debt from new regressions. Every run against a repo with legacy oversized files fails on the same findings, forcing repeated ad hoc waivers in the change brief instead of a real, checkable record of accepted debt.
- Issue #93: add an explicit, fail-closed structure-debt baseline so a repository can record "this file is already over budget by this much" once, have the checker suppress *exactly* that recorded amount, and force the record to move only in the tightening direction (fixed or worse-caught, never silently loosened).
- This is playbook infrastructure: the checker and its contracts (`project-structure`, `quality-gate`, `dev-workflow`) are copied into product repositories, so the default baseline path and validation rules must be safe and self-explanatory without this repo's context.

## Scope

### In scope
- `scripts/structure_rules.py` (new): file discovery/collection and per-file structure-rule analysis (`Finding`, `check_file`, `is_entrypoint`, `rust_inline_test_line_indices`, `collect_files`, `git_tracked_source_files`). No baseline knowledge.
- `scripts/structure_paths.py` (new): repo-relative baseline-path normalization (`normalize_entry_path`) and repo-root confinement (`path_escape`: rejects absolute/`..`/symlinked paths). No baseline schema or reconciliation knowledge.
- `scripts/structure_baseline.py` (new): baseline schema, validation, and reconciliation (`BaselineEntry`, `BaselineIssue`, `load_baseline_entries`, `validate_baseline_entries`, `reconcile_baseline`, `resolve_baseline_path`). No CLI/argparse knowledge; delegates path confinement to `structure_paths`.
- `scripts/check_structure.py`: explicit baseline file support (`.agents/structure-baseline.json` default, `--baseline PATH` override) wired through as thin CLI orchestration only — parses args, calls into `structure_rules`/`structure_baseline`, prints/exits. `--json` output separates blocking findings, accepted debt, and baseline errors.
- `tests/test_check_structure.py`: TDD coverage for the source-rule surface (rust inline-test detection, entrypoint detection, `check_file`).
- `tests/test_structure_baseline.py` (new): TDD coverage for every baseline acceptance criterion, invoked as a black-box subprocess against the CLI.
- `tests/test_structure_path_confinement.py` (new): TDD coverage for the five path-confinement/alias scenarios (absolute path, `..` traversal, symlinked leaf, symlinked parent directory, path-alias duplicate), invoked as a black-box subprocess against the CLI.
- `.agents/skills/project-structure/SKILL.md`: document the baseline as a ratchet, not a waiver.
- `.agents/skills/quality-gate/SKILL.md` + `references/quality-gate.md`: structural exit check (1b) records baseline-aware result; baseline errors and unmatched findings are `no-submit`, accepted debt is not "clean".
- `.agents/skills/dev-workflow/references/dev-workflow.md`: keep the structure-watch contract description consistent with the new baseline-aware output (only if the existing wording would otherwise mislead).
- `CHANGELOG.md`, this ExecPlan, `reports/workflow-contract-review/issue-93-structure-budget-ratchet.md`.

### Out of scope / non-goals
- Auto-generating or silently rewriting a baseline from current findings (no `--write-baseline`-style command in this change).
- Git-diff-only enforcement (baseline reconciliation is state-based, like the rest of the checker).
- Changing the canonical numeric limits (400 / 150 / 200).
- New dependencies (stdlib only, unchanged).
- Any product-repository baseline data.

## Constraints / Quality targets

- Stdlib-only; `make verify` stays green.
- No baseline file present (default path absent, `--baseline` not passed) → byte-identical CLI text and JSON output to the pre-change checker (AC2).
- Baseline validation must not depend on git history, mtimes, filename ordering, or network (AC5) — implemented via direct filesystem checks and explicit JSON, no git subprocess calls in the baseline path.
- Deterministic baseline-mode diagnostics: baseline errors sort by `(kind, path, rule)` and blocking/accepted findings by `(path, rule)`, independent of dict iteration order.

## Context & Orientation

- Key paths: `scripts/structure_rules.py` (source-rule collection/analysis), `scripts/structure_paths.py` (path normalization/confinement), `scripts/structure_baseline.py` (baseline schema/validation/reconciliation), `scripts/check_structure.py` (CLI wiring), `tests/test_check_structure.py` (source-rule unit tests), `tests/test_structure_baseline.py` (baseline black-box tests), `tests/test_structure_path_confinement.py` (path-confinement black-box tests), `.agents/skills/project-structure/SKILL.md`, `.agents/skills/quality-gate/SKILL.md` + `references/quality-gate.md`, `.agents/skills/dev-workflow/references/dev-workflow.md`.
- Existing behavior: `main()` collects git-tracked (or explicit) source files, runs the three canonical rules per file, prints `FINDING` lines or a `pass` message, exits `0` clean / `1` findings / `2` usage error.
- No script in this repo consumes `check_structure.py --json` output programmatically (`grep` sweep of the repo shows only prose/Markdown references), so the JSON shape may gain new top-level keys under baseline mode without breaking any in-repo consumer. The **no-baseline** JSON shape (a bare list) must not change, since it is the documented contract in the module docstring and referenced by `quality-gate`/`project-structure`.
- Convention: JSON config files under `.agents/` use kebab-case (`model-catalog.json`, `route-lockfile.json`) — the baseline default path follows this: `.agents/structure-baseline.json`.

## Design

### Module split (responsibility map)

The checker is split into four files, each with exactly one reason to change:

- `scripts/structure_rules.py` — **owns**: what counts as a source file, what counts as an entrypoint, how to count logic lines vs. inline tests vs. comments, and the three canonical rule checks. **Does not know** the baseline exists. Exports `Finding`, `check_file`, `is_entrypoint`, `rust_inline_test_line_indices`, `collect_files`, `git_tracked_source_files`, and the three `DEFAULT_MAX_*` constants. Change reason: a new rule, a new source extension, or a change to entrypoint/inline-test detection.
- `scripts/structure_paths.py` — **owns**: deciding whether a baseline-entry path string is safe to trust — pure alias normalization (`normalize_entry_path`) and repo-root confinement (`path_escape`: absolute path, `..` traversal, or a symlink anywhere in the path chain, all rejected). **Does not know** the baseline schema, `Finding`, or reconciliation exist; takes and returns plain strings only. Exports `normalize_entry_path`, `path_escape`. Change reason: a new class of unsafe path (e.g. a new traversal encoding) or a change to what "confined to the repo root" means.
- `scripts/structure_baseline.py` — **owns**: the baseline file schema (v1), loading/parsing, per-entry and cross-entry validation, and reconciling a baseline against a fresh set of `Finding`s *and this invocation's scanned-file scope* into `(blocking findings, accepted_debt, baseline_errors)`. **Does not know** about argparse, `sys.exit`, printing, or how a path is confined — it calls `structure_paths.normalize_entry_path`/`path_escape` and reports whatever they return as a `path-escape` issue. Exports `BaselineEntry`, `BaselineIssue`, `resolve_baseline_path`, `load_baseline_entries`, `validate_baseline_entries`, `reconcile_baseline`, `to_repo_relpath`, `DEFAULT_BASELINE_PATH`, `BASELINE_SCHEMA_VERSION`. Imports `Finding`/`SOURCE_EXTENSIONS` from `structure_rules` and `normalize_entry_path`/`path_escape` from `structure_paths`. Change reason: a new baseline-error kind, a schema version bump, or a change to what counts as "exact debt" (or, as this round showed, to what counts as "in scope" for staleness).
- `scripts/check_structure.py` — **owns**: CLI wiring only — `parse_args`, formatting/printing findings and baseline outcomes as text or JSON, `main()`'s exit-code decision, and the `__main__` guard. It imports from `structure_rules`/`structure_baseline` (not `structure_paths` directly — it has no reason to know path confinement exists), builds the `scanned_paths` scope from `collect_files()`'s result via `structure_baseline.to_repo_relpath` before calling `reconcile_baseline`, and re-exports `check_file`/`is_entrypoint`/`rust_inline_test_line_indices` via `__all__` so `tests/test_check_structure.py`'s existing `from scripts.check_structure import ...` keeps working unmodified. Change reason: a new CLI flag, a new output format, or a change to exit-code policy.

This mirrors the acceptance criteria directly: AC1–AC2 (baseline discovery/compatibility) and AC5 (no git/mtime/network dependency) belong to `structure_baseline.py`; AC3's path-confinement clause and the alias-normalization requirement belong to `structure_paths.py`; AC6/AC9 (CLI/JSON identity) belong to `check_structure.py`; the pre-existing three rules belong to `structure_rules.py` and are untouched by this change except for the file move.

### Import-ambiguity resolution (direct-script vs. package import)

Both `check_structure.py` and `structure_baseline.py` need to import their sibling module(s) (`structure_rules`, `structure_paths`, and in `check_structure.py`'s case, also `structure_baseline`). Two call shapes exist for this checker and both must keep working with no `scripts/__init__.py` added (this repo's `scripts/`/`tests/` are implicit PEP 420 namespace packages, not regular packages):

- **Package-qualified** (used by `tests/test_check_structure.py`'s `from scripts.check_structure import check_file`): the repo root is on `sys.path`, so `scripts` resolves as a namespace package and `from scripts.structure_rules import Finding` works.
- **Direct-script** (used everywhere else — skill docs, `dev-workflow`, `quality-gate`, and the subprocess-based tests in `tests/test_structure_baseline.py`: `python3 scripts/check_structure.py ...`): Python puts the script's own directory (`scripts/`) at `sys.path[0]`, not the repo root, so `scripts` is not importable from inside itself — only the unqualified `from structure_rules import Finding` resolves.

Rather than a try/except fallback (which hides which branch actually ran and can mask a real `ImportError` from an unrelated cause), each import site branches explicitly on `__package__`, which Python sets to `"scripts"` when imported package-qualified and to `""` (falsy) when run as a top-level script:

```python
if __package__:
    from scripts.structure_rules import Finding, SOURCE_EXTENSIONS
else:
    from structure_rules import Finding, SOURCE_EXTENSIONS
```

This is deterministic (one condition, one truth value per invocation mode) and self-documenting at the call site — a reader immediately sees both supported invocation shapes rather than inferring one from a caught exception.

### Baseline file contract

- Default path: `.agents/structure-baseline.json`, resolved relative to the current working directory (same root `collect_files`/`git_tracked_source_files` use). Optional `--baseline PATH` explicit override.
- If `--baseline` is passed and the path does not exist, this is a **usage error** (exit 2) — the caller asked for a specific file, not "auto-detect".
- If no `--baseline` is passed and the default path does not exist, behavior is **byte-identical** to before this change (AC2) — no baseline object is even constructed on that path.
- Schema (v1): `{"version": 1, "entries": [{"rule": str, "path": str, "value": int, "limit": int}, ...]}`. Exactly these keys, no extras, `rule` must be one of the three canonical rules, `value`/`limit` must be non-negative ints (bools rejected), `path` is a non-empty string interpreted relative to the working directory.
- Validation, in order, all fail-closed and collected (never stop at first issue), each becomes a `baseline-error` in output:
  1. Malformed top-level/entry schema (wrong types/keys/version) → `malformed-schema`.
  2. Entry's `path` escapes repo-root confinement — absolute, contains a `..` segment, or passes through a symlink anywhere in the chain from `Path.cwd()` to the target (even one resolving back inside the repo) → `path-escape` (checked by `scripts/structure_paths.py`'s `path_escape`, called from `structure_baseline.py`). Path aliases (`x.py`/`./x.py`) are normalized (`structure_paths.normalize_entry_path`) to one canonical string before this and every later check, so an alias cannot dodge duplicate detection.
  3. Entry names a rule outside the canonical three → `unknown-rule`.
  4. Two entries share `(rule, path)` → `duplicate-entry` (both excluded from matching — ambiguous).
  5. Entry's `path` does not exist on disk or is not a recognized source extension → `missing-path`.
  6. Entry's `limit` does not equal the canonical limit currently in effect for that rule (from CLI flags/defaults) → `threshold-mismatch`.
  7. A finding for `(rule, path)` has `value` *greater* than the baseline's recorded `value` → `regression` (worse than recorded, not accepted debt).
  8. A finding for `(rule, path)` has `value` *less* than the baseline's recorded `value`, or a **scanned** baseline entry no longer reproduces any finding at all → `stale-baseline` (file improved; the looser baseline must be refreshed so debt is never overstated — this is AC4). Scope only ever changes this one outcome: a schema-valid, path-valid, canonical-limit-valid baseline entry for a path *outside this invocation's scanned scope* (see below) that matches no finding is left neutral instead of `stale-baseline` — it is not scanned, so nothing can be concluded about whether the file still breaches the rule. Kinds 1–6 above are baseline-document defects checked against every parsed entry unconditionally in `validate_baseline_entries`/`_parse_baseline_entry`, before `scanned_paths` even exists as a concept; they always fail closed on every invocation, partial or full, whether or not the entry's file was touched — only kind 8's "no longer reproduces a finding" sub-case is scope-dependent.
- **Scanned scope**: `reconcile_baseline` takes a `scanned_paths` set — the cwd-relative posix paths of every file `check_structure.py` actually scanned this invocation (all git-tracked source files on a default no-args scan; only the explicit files/dirs given otherwise). This closes a bug found by independent review: without it, an untouched baseline entry for a file *not* among the files a partial, explicit-paths invocation checked was indistinguishable from a genuinely stale one, so any unrelated touched-file check could fail merely because some other, uninvolved file had valid recorded debt. A full default scan's scope is every tracked source file, so it is unaffected and still detects every stale entry in the repository.
- A finding whose `(rule, path)` has **no** baseline entry is an ordinary blocking finding — new debt is never silently accepted, matching AC2/AC3.
- A finding whose `(rule, path)` matches a baseline entry **exactly** (`value == entry.value` and canonical `limit == entry.limit`, which is already guaranteed by validation step 5) is `accepted_debt`: it does not block, but it is never reported as "clean"/"pass" in the same breath as zero-finding-zero-debt.
- Exit code: `0` only when there are zero blocking findings and zero baseline errors (accepted debt alone does not fail); `1` when either exists; `2` only for CLI usage errors (bad flags, explicit `--baseline` path missing).

### Testing strategy

- Unit tests in `tests/test_check_structure.py` (source-rule surface: rust inline-test detection, entrypoint detection, `check_file`), `tests/test_structure_baseline.py` (TDD, Red → Green → Refactor per case, run as black-box subprocess invocations of the real CLI: absent-baseline compatibility (text + JSON), exact-debt acceptance, regression, improvement/stale baseline (both directions: value-decreased-but-still-over-limit, and fully-fixed-so-no-finding-at-all), an untouched-entry-is-neutral-on-partial-scan case and a full-default-scan-still-detects-staleness case (the scanned-scope fix), an untouched-integrity-errors-still-fail-closed case proving `missing-path`/`threshold-mismatch` are never scope-limited even when their entry's file is not touched (the Director-requested clarification), duplicate entries, malformed schema (multiple variants), missing/non-source path, unknown rule, threshold mismatch, new finding without baseline entry, explicit `--baseline` override, explicit `--baseline` missing file (usage error), JSON output shape under baseline mode), and `tests/test_structure_path_confinement.py` (same black-box subprocess style: absolute path, `..` traversal, symlinked leaf, symlinked parent directory, and path-alias duplicate detection).
- Manual verification: run the checker against its own touched files with and without a baseline, confirm exit codes and diagnostics read as actionable.

## Surprises & Discoveries

- **Self-referential structure-budget breach.** Implementing the baseline feature grew `scripts/check_structure.py` from 287 to 759 lines and `tests/test_check_structure.py` from ~112 to 406 lines in the first (verbose, fully-tested) pass — both over the checker's own 400-line `source-file-lines` limit that this same change enforces. The first-pass file-ownership allowlist had no new-file entries, so the initial fix was readability-preserving compaction only (helper extraction, single-line argparse/dataclass declarations, condensed f-strings, single blank line between top-level defs), landing at 395/394 lines — under budget, all tests green, but at the cost of Black/PEP8 readability norms this repo otherwise follows by convention (no linter enforces blank-line spacing, so this was a style tradeoff, not a lint violation).
- **User review rejected the compaction-only fix.** The reviewer judged that dense one-liners and long signatures violate the intent of the structure gate and the `project-structure` rule that a genuine budget breach is a signal to split responsibilities, not to compact them into invisibility — compaction under a size limit is a way to *hide* the same amount of complexity from the gate that is supposed to surface it. The reviewer expanded the file-ownership allowlist to include `scripts/structure_rules.py`, `scripts/structure_baseline.py`, and `tests/test_structure_baseline.py` and required a real module split. This is the correct resolution: the first-pass Decision log entry about "compaction over splitting because splitting was out of scope" is superseded below, not merely appended to, since its stated constraint (no new files allowed) no longer holds and its conclusion should not be read as if it still applies.

### Responsibility summary

- `scripts/structure_rules.py`: single responsibility — decide what a "finding" is for a given file, with no awareness that a baseline concept exists. Change reason: a new rule, a new source extension, or a change to entrypoint/inline-test detection.
- `scripts/structure_paths.py`: single responsibility — decide whether a path string is safe to trust as a baseline entry's `path` (confined to the repo root, symlink-free) and normalize aliases, with no awareness that a baseline schema or `Finding` concept exists. Change reason: a new class of unsafe path, or a change to what "confined to the repo root" means.
- `scripts/structure_baseline.py`: single responsibility — validate a baseline file's shape and reconcile it against a set of findings, with no awareness of argparse, printing, exit codes, or how a path is confined (delegates that to `structure_paths`). Schema/duplicate/path/threshold validity is checked against every entry unconditionally; only the narrower "does this entry's file still breach the rule" reconciliation question is scoped to the current invocation's scanned files. Change reason: a new baseline-error kind, a schema version bump, or a change to what counts as "exact debt" or "in scanned scope".
- `scripts/check_structure.py`: single responsibility — CLI wiring (parse args, call into the other modules, print, decide the exit code). It does not become a second tool bolted onto the first; it is fully inert with respect to baseline reconciliation when no baseline is present (AC2).
- `project-structure`, `quality-gate`, and `dev-workflow` docs each describe the same contract from their own vantage point (generative layout guidance, final submit gate, workflow record template) without duplicating the schema definition itself — the schema lives once, in `scripts/structure_baseline.py`'s module docstring and constants.
- Baseline *authorship* (deciding what debt to accept, at what value) is explicitly the maintainer's responsibility, never the tool's — the tool only validates and reconciles what a human wrote.
